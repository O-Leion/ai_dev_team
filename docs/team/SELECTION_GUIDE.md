# チーム選定ガイドライン (Team Selection Guide)

プロジェクト開始時に、要件定義（Inception）フェーズで以下の基準に従ってメンバーを選定します。

## 1. 最小構成 (Core Team)
小規模なプロトタイプや、検証フェーズでは以下の2名、あるいは1名が兼務して開始します。
- **PL (Project Leader):** 技術選定、実装、設計を兼務。
- **PM (Project Manager):** 要件整理、スケジュール管理。
*※必要に応じて、PLがPMを兼務することもあります。*

## 2. 要件別追加メンバーの基準
開発するアプリケーションの特性に応じて、以下の役割をアサインします。

| プロジェクト特性 | 追加すべき役割 | 理由 |
| :--- | :--- | :--- |
| **UI/UXが重要なWeb/アプリ** | **DS, FE** | ユーザー体験の質と、フロントエンドの複雑性に対応するため。 |
| **データ保持・複雑なロジック** | **BE** | データベース設計とセキュアなAPI構築が必要なため。 |
| **AI/LLMを活用する機能** | **AI** | プロンプト調整やRAGの精度管理に専門知識が必要なため。 |
| **インフラ負荷が高い・CI/CD重視** | **Ops** | 自動化されたデプロイフローとスケーラビリティを担保するため。 |
| **機密情報・決済・広範囲の公開** | **Sec** | データ漏洩リスクの低減と、堅牢な認証・認可が必要なため。 |
| **商用リリース・高品質の追求** | **QA** | 網羅的なテストとリリース判定の客観性を保つため。 |

## 3. 選定プロセス
1. **要件整理:** PM/PLがアプリの目的と機能をリストアップ。
2. **役割マッピング:** 上記の基準に基づき、必要な役割を決定。
3. **アサイン:** 決定した役割に基づき、具体的なメンバー（またはAIエージェントのペルソナ）を割り当て。

## 4. 構成例
- **例1：AIチャットツール (PoC)**
  - PL, AI, FE (3名)
- **例2：シンプルなランディングページ**
  - PM/PL, DS, FE (3名)
- **例3：大規模業務システム (商用)**
  - 全ての役割 (8名)
