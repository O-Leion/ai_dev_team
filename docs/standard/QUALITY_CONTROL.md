# 品質管理ガイドライン (Quality Control Guideline)

本プロジェクトチームは、継続的な静的解析と多層的なテストにより、高いコード品質と保守性を担保します。

## 1. 静的解析 (Static Analysis)
コードを動かす前に、ツールによる自動チェックを必須とします。

- **Linting:** プロジェクトごとに最適なLinter（ESLint, Ruff, GoLint等）を使用し、コーディング規約への準拠を確認。
- **Type Checking:** TypeScript, Python (mypy/pyright), Go 等の型システムを活用し、実行前エラーを最小化。
- **Formatting:** Prettier, Black, gofmt 等により、チーム全体で統一されたコードスタイルを維持。

## 2. テスト戦略 (Testing Strategy)
以下の4層で品質を検証します。

| テスト層 | 責任者 | 内容 |
| :--- | :--- | :--- |
| **Unit Test** | FE/BE/AI | 関数・クラス単位の動作確認。正常系・異常系の網羅。 |
| **Integration Test** | BE/FE | APIとDBの連携、複数コンポーネント間の動作確認。 |
| **E2E Test** | QA/FE | ユーザーの操作シナリオに基づいた画面全体の動作確認（Playwright等）。 |
| **Security Scan** | Sec | ライブラリの脆弱性チェック、シークレット漏洩の自動検知。 |

## 3. 品質担保のプロセス (Automation)
- **Pre-commit Hook:** コミット前に、静的解析と重要な単体テストをローカルで実行。
- **CI (Continuous Integration):** プルリクエスト作成時に、全ての静的解析とテストを自動実行。パスしない限りマージ不可とする。
- **Manual Review:** 機械的なチェックに加え、PLによるコードレビューで設計の妥当性を確認。

## 4. プロジェクトごとの柔軟な適用
個人開発やプロトタイプ初期フェーズでは、クライアントの承諾を得て「静的解析のみ」に絞り、スピードを優先することも可能です。ただし、商用リリース前には必ず網羅的なテストを実施します。
