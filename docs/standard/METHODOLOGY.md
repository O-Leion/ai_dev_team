# 開発手法ガイドライン (Development Methodology)

本プロジェクトチームは、個人開発のスピード感と品質を両立させるため、**「ハイブリッド・スクラム」**を標準手法として採用します。ただし、クライアント（ユーザー）の指定がある場合は、プロジェクトごとに最適な手法へ柔軟に変更します。

## 1. 標準手法：ハイブリッド・スクラム (Default)

### A. 準備フェーズ (Waterfall-like)
- **目的:** 全体のゴール共有と土台作り。
- **タスク:** 
  - 要件定義（PM/PL）
  - システムアーキテクチャ設計（PL）
  - セキュリティ・インフラ基本設計（Sec/Ops）
- **成果物:** 要件定義書、構成図、初期バックログ（タスク一覧）。

### B. 実行フェーズ (Scrum-like)
- **サイクル:** 1〜2週間の「スプリント」単位で進行。
- **スプリントの流れ:**
  1. **プランニング:** 今週作る機能をバックログから選定。
  2. **開発・テスト:** FE/BE/AI/QAが連携して実装。
  3. **デモ・レビュー:** 完了した機能をクライアントに提示し、フィードバックを得る。
  4. **ふりかえり:** 次のサイクルをより良くするための調整。

## 2. 手法の柔軟な変更 (Flexibility)

プロジェクト開始時に、クライアントの要望や特性に応じて以下の選択肢から手法を決定します。

| 手法 | 適したケース | 特徴 |
| :--- | :--- | :--- |
| **ハイブリッド (Default)** | 通常のアプリ開発 | 設計の安心感と、変更への強さを両立。 |
| **純粋なスクラム** | 実験的な開発、R&D | 計画を最小限にし、即座にプロトタイプ作成。 |
| **ウォーターフォール** | 仕様が完全に固定された開発 | 全工程のドキュメント化と厳格な進捗管理。 |
| **カンバン (Kanban)** | 保守・運用、小規模改善 | サイクルを決めず、優先順位の高い順に消化。 |

## 3. クライアントとの関わり
- **意思決定:** 手法の最終決定権はクライアントにあります。
- **報告形式:** 選定された手法に基づき、PM/PLが最適な形式（週報、デモ会、チャット報告など）で進捗を共有します。
